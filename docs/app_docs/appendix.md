# Appendix: cluster_summary_viewer パイプライン詳細

## 1. ETL パイプライン

### 1-1. OSM fetch

Overpass API で日本全国の温泉関連ノードを取得する。

**取得タグ条件:**

```
node["natural"="hot_spring"]
node["natural"="spring"]["hot_spring"="yes"]
node["amenity"="public_bath"]
node["amenity"="spa"]
node["amenity"="bathhouse"]
node["leisure"="hot_spring"]
node["leisure"="bath"]
node["leisure"="bathing_place"]
node["tourism"="spa"]
node["tourism"="spa_resort"]
```

- エリア絞り込み: `area["name"="日本"]["admin_level"="2"]`
- **レコード数:** 2,544ノード

---

### 1-2. AEF fetch

Google Earth Engine (GEE) の AlphaEarth Foundations (AEF) コレクション `GOOGLE/SATELLITE_EMBEDDING/V1/ANNUAL` から、各ノード周辺の埋め込みベクトルをサンプリングしてエクスポートする。

**パラメータ（最終実行時）:**

| パラメータ | 値 |
|---|---|
| コレクション | `GOOGLE/SATELLITE_EMBEDDING/V1/ANNUAL` |
| 対象期間 | 2024-01-01 〜 2024-12-31 |
| バッファ半径 | 100m |
| サンプルスケール | 10m |
| サンプル数/ノード | **5点** |
| シード | 42 |

- 各行: `node_id`, `A00`〜`A63`（64次元 AEF ベクトル）
- **レコード数:** 12,419行（2,521ノード × 約5行）

---

### 1-3. Embed

AEF 64次元ベクトルを UMAP で 10次元に削減する。

**パラメータ:**

| パラメータ | 値 |
|---|---|
| 入力次元 | 64 |
| 出力次元 | 10 |
| metric | euclidean |
| n_neighbors | 15（デフォルト）|
| random_state | 42 |

> **Note:** metric に euclidean を指定しているが、AEF ベクトルは L2 正規化済みのため、ユークリッド距離とコサイン距離は単調に対応し実質同等になる。

- **レコード数:** 12,419行（入力と同数）

---

### 1-4. Cluster

UMAP 10次元埋め込みに HDBSCAN を適用してクラスタラベルを付与する。

**パラメータ:**

| パラメータ | 値 |
|---|---|
| min_cluster_size | 10 |
| min_samples | 10 |

**結果:**

| 指標 | 値 |
|---|---|
| クラスタ数 | 134 |
| ノイズ率 | 58.2%（7,226 / 12,419行、行レベル）／ユニークノードでは約 18.7%（471 / 2,521）|

---

## 2. Analysis パイプライン

### 2-1. fetch GSI

国土地理院・J-SHIS の公開 API から各ノードの標高と地形分類を取得する。

**取得元:**

| API | 使用フィールド |
|---|---|
| 国土地理院 標高API | 標高（m）|
| J-SHIS 表層地盤情報API | 地形分類名 |

- レート制限: 1秒スリープ
- **レコード数:** 2,050ノード

---

### 2-2. wiki

Wikipedia geosearch API でノード周辺の記事を取得し、フィルタ・要約する。

**fetch**

| パラメータ | 値 |
|---|---|
| 検索半径 | 1000m |
| 記事数上限/ノード | 10件 |

- **レコード数:** 2,544ノード、ユニーク記事 10,292件

**filter**

全文から以下のセクションを除去（見出し名の部分一致）:  
`脚注` / `外部リンク` / `関連項目` / `出典` / `注釈` / `参考文献` / `歴史` / `沿革`

ノイズノードの wiki を除外し、クラスタ割当済みノード（2,050）に紐づく記事のみを要約対象とする。

- **レコード数:** 10,292 → 8,590記事

**summarize**

Snowflake `AI_COMPLETE` で記事ごとに要約を生成。途中再開対応。プロンプト詳細は [3-1](#3-1-wikipedia-要約) 参照。

| パラメータ | 値 |
|---|---|
| モデル | `claude-sonnet-4-6` |
| 目安文字数 | 400字 |
| 抽出方針 | 場所・地域・施設の特徴（種別・立地・規模・特色）に絞る。人物・事象・歴史は除外 |

- **レコード数:** 8,590行

---

### 2-3. fetch OSM surroundings

Overpass API で各ノード周辺 100m の OSM 要素（node/way/relation）を取得する。

| パラメータ | 値 |
|---|---|
| 半径 | 100m |
| クエリタイプ | nwr（node/way/relation） |
| way・relation の座標 | out center（重心座標を利用）|
| バッチサイズ | 50ノード/リクエスト |

- **レコード数:** 2,544ノード

---

### 2-4. dendrogram

HDBSCAN クラスタの UMAP 重心ベクトルから階層クラスタリングを行う。

**手順:**

1. 各クラスタの UMAP 重心ベクトル（10次元平均）を計算
2. 重心間コサイン距離行列を計算
3. Average linkage で階層クラスタリング
4. Nominatim で各クラスタ重心の逆ジオコーディング（行政区名取得）

**統計指標:**

| 指標 | 定義 |
|---|---|
| σUMAP | クラスタ内ノードの UMAP 各次元 std の平均（AEF 空間均質性）|
| σgeo (km) | `√(lat_std_km² + lon_std_km²)`（地理的広がり）|

- **レコード数:** クラスタ統計 134件、linkage 133行

---

### 2-5. select pairs

樹形図からクラスタペアを選定する。

**樹形図リーフペア**
樹形図末端で直接繋がる2クラスタのペア（リーフペア）を抽出。リーフペアを持たない孤立リーフは、UMAP 重心コサイン距離が最小のクラスタとペアリング（孤立リーフペア）。

**ペア統計:**

| 指標 | 定義 |
|---|---|
| cosine_distance | 両クラスタの UMAP 重心間コサイン距離 |

- **レコード数:** 84組（リーフペア49 + 孤立リーフペア35）

---

### 2-6. geotext_summary

Snowflake `AI_COMPLETE` でクラスタ特徴仮説とペア考察を生成する。

**node_text 構築:**

ノードごとに以下を結合する。

```
【地形】標高XXXm、地形分類: XXXX
【周辺施設】
amenity=hot_spring
...
【周辺Wikipedia】
（要約テキスト）
```

**クラスタ特徴仮説**

node_text からクラスタの本質的な特徴を推察する。温泉地としての側面と地形・土地利用としての側面の2バリアントを生成。

**クラスタペア考察**

クラスタ特徴仮説をもとに、ペアを構成する2クラスタの共通点と相違点を考察する。

**パラメータ:**

| パラメータ | 値 |
|---|---|
| モデル | `claude-sonnet-4-6` |
| サンプル上限/クラスタ | 50ノード |

**ステップ:**

| ステップ | 内容 |
|---|---|
| step_a | node_text を Snowflake にアップロード |
| step_b | クラスタ特徴仮説を生成（プロンプト → [3-2](#3-2-クラスタ特徴温泉サマリー), [3-4](#3-4-クラスタ特徴地理サマリー)）|
| step_d | クラスタペア考察を生成（プロンプト → [3-3](#3-3-クラスタペア考察温泉), [3-5](#3-5-クラスタペア考察地理)）|

- 途中再開対応: 処理済みクラスタ・ペアはスキップ
- **レコード数:** クラスタ特徴 134行、クラスタペア 84行（各バリアント共通）

**クラスタ特徴仮説のプロンプト入力文字数（プロンプト719字込み）:**

| | 文字数 |
|---|---|
| 最大 | 289,092 |
| 中央値 | 28,778 |
| 平均 | 47,196 |

- 平均クラスタサイズが22ノードのため、50件上限に達するケースは少なく中央値は約2.9万字
- 最大29万字はノード数185件・ノードあたり平均5,500字超のクラスタ（Wikipedia記事が充実した観光地集中クラスタ）が50件サンプリングされた結果

## 3. 使用プロンプト

### 3-1. Wikipedia 要約

```
以下のWikipedia記事を400字を目安に要約してください。
場所・地域・施設の特徴（種別、立地、規模、特色など）を簡潔にまとめること。
人物・事象・歴史など土地の特徴に当たらない記述は省くこと。
冗長な表現や接続詞の繰り返しを避け、客観的かつ情報密度の高い文体で記述すること。

【記事】
```

---

### 3-2. クラスタ特徴（温泉サマリー）

```
# Role
あなたは高度な地理空間データアナリストであり、日本の温泉地研究の専門家です。

# context
このクラスタ分析は、AlphaEarth Foundations が提唱する地球環境データの統合的活用の一環です。
1. AEFデータ基盤: 地形、植生、気候、土地利用など、地球表面の多層的な情報を統合したデータセットに基づいています。
2. UMAP + HDBSCAN: 高次元特徴の類似性に基づき、単なる地理的近接性ではなく「物理的性質の似た地点」を密度ベースで抽出しています。

# Instructions
以下の2つの問いに、AEFとは別に提供されたノード情報に基づいて答えてください。

1. core_trait
   このクラスタを端的に表す最も本質的な温泉地や温泉施設としての特徴は何か。
   「山岳型」「秘湯型」など一般的な分類ではなく、このクラスタ固有の性質を鋭く一言で指摘すること。
   core_traitを支持する根拠はノード情報のどこに見えるか。

2. variation
   このクラスタで多様な特徴が見られる温泉地や温泉施設としての側面は何か。
   地域、土地利用形態、地形、地質、周辺施設に関するミクロとマクロの視点から検討すること。
   variationを支持する根拠はノード情報のどこに見えるか。

# Constraints
- 回答は必ず指定された JSON 形式のみを出力してください。
- 各回答は、100文字〜500文字程度の具体的な文章で記述してください。
- 専門用語（地形分類名など）は正確に使用してください。

【ノード情報（"---" 区切り）】
```

**出力スキーマ:** `{core_trait, variation}`

---

### 3-3. クラスタペア考察（温泉）

```
# Role
あなたは高度な地理空間データアナリストであり、日本の温泉地研究の専門家です。

# context
（クラスタ特徴と同じ AEF/UMAP+HDBSCAN の説明）

# Instructions
提供された2クラスタの特徴情報を比較し、以下の2つの切り口で考察してください。

1. similarity
   両クラスタが共有する温泉地としての本質的な特徴は何か。

2. contrast
   両クラスタを分ける最も本質的な違いは何か。

# Constraints
- 回答は必ず指定された JSON 形式のみを出力してください。
- 各回答は、30文字〜150文字程度の具体的な文章で記述してください。
- 専門用語（地形分類名など）は正確に使用してください。
```

**入力:** 各クラスタの `core_trait`, `variation` を結合して投入  
**出力スキーマ:** `{similarity, contrast}`

---

### 3-4. クラスタ特徴（地理サマリー）

クラスタ特徴（温泉サマリー）との差分:

- Role: 「温泉地研究の専門家」の記述を削除
- `core_trait` / `variation` の問い文: 「温泉地や温泉施設としての」の記述を削除
- `core_trait` の例: 「山岳型」→「山岳地帯」「都市部」
- **Constraints に追加:** `いずれについても温泉地・温泉施設以外の側面に着目して回答すること。`

**出力スキーマ:** `{core_trait, variation}`

---

### 3-5. クラスタペア考察（地理）

クラスタペア考察（温泉）との差分:

- Role: 「温泉地研究の専門家」の記述を削除
- `similarity`: 「温泉地としての」→「土地としての」
- **Constraints に追加:** `いずれについても温泉地・温泉施設以外の側面に着目して回答すること。`

**出力スキーマ:** `{similarity, contrast}`
